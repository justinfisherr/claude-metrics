# Jazz Taste ML — Claude Guide

This repo holds Justin's personal jazz recommendation ML pipeline: training, prediction, and an interactive dashboard.

## What This Is

A self-improving system that learns Justin's jazz taste from rated tracks and recommends new ones. The training data lives in his Obsidian vault (`~/Documents/remote/Music/Jazz Dataset/training-data.md`), not in this repo.

## Repo Layout

- `train.py` — Reads training data, engineers features, trains Ridge + Random Forest models, outputs `model.joblib` and `dashboard-data.json`
- `predict.py` — Scores candidate tracks against the trained model. Accepts `--json` or `--file` input.
- `model.joblib` — Trained model artifact (Random Forest, currently best)
- `dashboard-data.json` — Generated by `train.py`, consumed by `index.html`
- `index.html` — Single-page dark-themed dashboard (Chart.js). Reads `dashboard-data.json` at load.
- `requirements.txt` — Python dependencies

## Key Workflows

### Retrain the model
```bash
python3 train.py
```
Reads from the Obsidian vault training data, outputs updated `model.joblib` and `dashboard-data.json`. Always retrain after new tracks are added to the dataset.

### Score a candidate track
```bash
python3 predict.py --json '{"title":"Track","artist":"Artist","energy":5,...}'
```
See `predict.py` header for full usage. The JSON must include all fields the model expects (energy, tempo, moods, subgenres, instrumentation, audio_features, etc).

### View the dashboard
Open `index.html` in a browser (or serve locally with `python3 -m http.server`). It fetches `dashboard-data.json` from the same directory.

## Feature Engineering

The model uses ~75 features across these categories:
- **Core numeric:** energy, year, tempo, harmonic_complexity, replayability
- **Era:** one-hot encoded
- **Instrument:** primary instrument group one-hot
- **Moods:** common moods one-hot + mood_count
- **Subgenres:** common subgenres one-hot + subgenre_count
- **Ensemble:** ensemble_size, has_guitar, has_strings, has_vocals, is_pianoless, has_trombone
- **Label:** common labels one-hot
- **Key players:** key_player_count
- **Discovery:** source one-hot (self, claude-recommendation, autoplay)
- **Engagement:** has_favorite_moments, replayability
- **Interactions:** energy × tempo
- **Artist:** artist_mean_rating (LOO), artist_track_count
- **Audio:** duration_s, popularity, tempo_bpm, time_signature, is_live

Dataset is small (~62 tracks), so avoid features that add noise without signal. LOO encoding is used for anything derived from ratings.

## Dashboard

The dashboard (`index.html`) is a single-page app using Chart.js with a dark theme. Current panels:
- Model Performance, Taste Clusters (PCA), Feature Importance, Predicted vs Actual
- Rating Distribution, Cluster Profiles (radar), Correlations (heatmap)
- Taste Constellation (Canvas 2D), Circle of Fifths (Canvas 2D)
- Your Taste vs The World (popularity scatter), Artist Journeys
- Mood × Rating (heatmap, collapsible), Replayability vs Rating
- Hall of Fame (collapsible, limit 20), Year vs Rating, Duration vs Rating
- Rating by Instrument (with track list dropdown), Label Breakdown (with track list dropdown)
- Biggest Misses, Top Artists, Era Breakdown, Harmonic Complexity, Discovery Source, Sound Profile

When adding visualizations:
- Match the dark-theme CSS variables (see `:root` block)
- Use `panel span-6` or `panel span-12` grid layout
- Add a nav link in the `<nav class="section-nav">` block
- Wire the render function in the fetch chain

## Rules

- **Don't edit training-data.md from this repo.** That's managed by the Obsidian-side agents.
- **Update taste-profile.md and model-notes.md after every log session.** Whenever new tracks are logged to training-data.md, the taste profile (`~/Documents/remote/Music/Jazz Dataset/taste-profile.md`) and model notes (`~/Documents/remote/Music/Jazz Dataset/model-notes.md`) must also be updated with any new patterns, artist insights, or preference shifts. These files are what recommendation agents read — stale profiles lead to bad recommendations.
- **Always commit `model.joblib` and `dashboard-data.json` together** after retraining — they're a matched pair.
- **Push to `main` branch** on `justinfisherr/claude-metrics`.
- **Test the dashboard locally** (`python3 -m http.server`) before pushing HTML changes.
- **Back up training-data.md to git after every log.** Whenever new tracks are logged to `~/Documents/remote/Music/Jazz Dataset/training-data.md`, copy it into this repo and commit+push. This ensures we always have a versioned backup. The file was previously untracked and got corrupted with no way to recover except from Claude's file-history snapshots — never again.
  ```bash
  cp ~/Documents/remote/Music/Jazz\ Dataset/training-data.md ~/jazz-ml/training-data.md
  cd ~/jazz-ml && git add training-data.md && git commit -m "Backup training data (N tracks)" && git push
  ```
